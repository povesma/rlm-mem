## Problem Statement

The `/dev:*` command files hardcode behavioral rules — code style
(120 char lines, no comments, "handler" terminology), testing approach
(TDD, test-external-only), workflow strictness (docs-first, correction
capture), and tool selection (RLM enabled, claude-mem required). These
defaults work for the author's projects but are wrong for many others:

- A Python data science project needs different style rules than a
  TypeScript API
- A solo prototyping session needs less docs-first strictness than a
  team production feature
- A project using Playwright MCP needs different subagent declarations
  than one without
- A user who prefers Cline memory bank over claude-mem can't switch

**Who**: Developers using `/dev:*` across different projects and tasks
**What**: No way to customize workflow behavior per project or context
**Why**: Either live with wrong defaults or manually edit command files
**Future context**: Profiles are the prerequisite for a dynamic
recommendation system where `/dev:start` suggests the right profile
based on repo analysis

---

## Research Findings

(Full report: `tasks/012-PROFILES-research-report.md`)

Claude Code already provides the primitives we need:

1. **`.claude/rules/*.md`** with YAML frontmatter `paths:` scoping —
   behavioral instructions that load into context
2. **`settings.json`** — hard enforcement (permissions, hooks, MCPs)
3. **Skills** — load-on-invoke only, don't pollute context
4. **`CLAUDE.md` hierarchy** — user → project → subdirectory layering

**Key constraints from research:**
- Rules >200 lines reduce LLM adherence significantly
- Prompt-based config is best-effort, not guaranteed
- Hard enforcement (tool blocking, MCPs) must use `settings.json`
- Specific, verifiable instructions work; vague ones get ignored

**Implication**: A profile is not a new config engine — it's a
curated composition of existing Claude Code primitives.

---

## What Is a Profile

A profile is a named configuration bundle containing:

1. **Rules** (`.md` files) — behavioral instructions injected into
   LLM context (code style, testing, workflow strictness)
2. **Tool declarations** (YAML) — which tools are enabled/disabled
   (RLM, memory backend, test subagents)
3. **MCP declarations** (YAML) — which MCP servers the profile
   expects (context7, playwright, claude-mem)

A profile does NOT replace `CLAUDE.md` or `settings.json` — it
layers on top of them. The user's personal `~/.claude/CLAUDE.md`
and project's `CLAUDE.md` remain untouched.

## Goals

### Primary
- Define a profile schema (YAML) that declares rules, tools, and MCPs
- Provide a `/dev:profile` command to activate a named profile
- Ship 2-3 built-in profiles (quality, fast, minimal) as examples
- `/dev:health` validates that active profile's MCP requirements are met

### Secondary
- Profiles storable at user scope (`~/.claude/profiles/`) and project
  scope (`.claude/profiles/`)
- Project profiles override user profiles (most-specific-wins)
- Command files read active profile at runtime instead of hardcoded
  values

## Target State (V1)

```
~/.claude/profiles/
├── quality.yaml         # built-in: full workflow
├── fast.yaml            # built-in: speed over ceremony
└── minimal.yaml         # built-in: no RLM, no memory

.claude/profiles/        # project-specific (optional)
└── my-project.yaml      # project overrides

~/.claude/active-profile.yaml  # symlink or copy of active profile
```

Profile YAML schema:

```yaml
name: quality
description: Full workflow — docs-first, TDD, RLM + claude-mem

rules:
  code_style:
    line_length: 120
    comments: minimal
    naming_convention: handler
  testing:
    approach: tdd
    scope: [unit, integration]
    subagents: [test-backend, test-review]
  workflow:
    docs_first: strict
    correction_capture: true
    scope_drift: warn

tools:
  rlm: true
  memory_backend: claude-mem    # claude-mem | none
  # future: cline-memory-bank, custom

mcps:
  required: [context7]
  optional: [playwright]
```

Command files read `~/.claude/active-profile.yaml` at runtime and
adapt behavior accordingly. If no profile is active, built-in defaults
(equivalent to current hardcoded behavior) apply.

## User Stories

### Story 1 — Activate a named profile

**As a** developer starting a session,
**I want** to run `/dev:profile use quality` to activate a profile,
**so that** all `/dev:*` commands use those settings.

**Acceptance Criteria:**
- [ ] `/dev:profile use <name>` loads a profile from
  `~/.claude/profiles/<name>.yaml` or `.claude/profiles/<name>.yaml`
- [ ] Active profile written/symlinked to
  `~/.claude/active-profile.yaml`
- [ ] Confirmation message shows profile name and key settings
- [ ] If profile not found, error with list of available profiles

### Story 2 — List available profiles

**As a** developer,
**I want** to run `/dev:profile list` to see all profiles,
**so that** I know what's available.

**Acceptance Criteria:**
- [ ] Lists profiles from both `~/.claude/profiles/` and
  `.claude/profiles/` (if present)
- [ ] Shows name, description, and source (user/project) for each
- [ ] Marks the currently active profile

### Story 3 — Command files respect active profile

**As a** command file (`impl.md`),
**I want** to read `~/.claude/active-profile.yaml` and adapt my
behavior,
**so that** code style, testing, and workflow rules match the profile.

**Acceptance Criteria:**
- [ ] `impl.md` reads active profile at start of implementation
- [ ] Code Style section uses profile's `line_length`, `comments`,
  `naming_convention` values
- [ ] Testing section uses profile's `approach` and `subagents`
- [ ] Workflow section uses profile's `docs_first` and
  `correction_capture` settings
- [ ] If no active profile, uses current hardcoded defaults

### Story 4 — Health check validates profile MCPs

**As a** developer who activated a profile,
**I want** `/dev:health` to verify that required MCPs are available,
**so that** I know if my environment matches the profile's needs.

**Acceptance Criteria:**
- [ ] `health.md` reads active profile's `mcps.required` list
- [ ] For each required MCP, checks if the MCP server is responsive
- [ ] Reports pass/fail per MCP with fix suggestions
- [ ] Optional MCPs listed as warnings, not failures

### Story 5 — Built-in profiles shipped

**As a** new user,
**I want** 2-3 built-in profiles installed by default,
**so that** I have useful starting points.

**Acceptance Criteria:**
- [ ] `quality.yaml` — full workflow (current defaults): docs-first
  strict, TDD, RLM + claude-mem, context7 required
- [ ] `fast.yaml` — speed mode: docs-first relaxed, test-after,
  RLM enabled, no correction capture
- [ ] `minimal.yaml` — bare bones: no RLM, no memory backend, no
  docs-first, no subagents
- [ ] `install.sh` copies profiles to `~/.claude/profiles/`

### Story 6 — Container-based testing environment

**As a** developer working on the profile system,
**I want** a Docker-based test environment with Claude Code CLI,
**so that** I can test profile changes without affecting my live
`~/.claude/` installation.

**Acceptance Criteria:**
- [ ] Dockerfile installs Claude Code CLI
- [ ] Auth persists across container restarts (volume-mounted)
- [ ] Container has a clean `~/.claude/` with only this package
  installed
- [ ] Can run `/dev:start` and `/dev:profile` inside the container

## Out of Scope (V1)

- **Profile inheritance/extends** (`extends: default`) — V2
- **Auto-detection** (`/dev:start` recommends profile) — V2
- **On-the-fly profile creation** — V2
- **Per-feature profiles** (different profile per PRD) — V2
- **Memory backend switching** (claude-mem → Cline memory bank) — V2,
  requires adapter layer
- **Settings.json patching** (profile auto-configures MCPs) — V2,
  risky to automate
- **Profile validation CLI** (lint a profile YAML) — nice to have

---

## Technical Constraints

- Profile YAML is read by Claude (the LLM) at runtime via a `Read`
  tool call in the command prompt — not preprocessed or injected
- This means profile instructions are best-effort (LLM may not follow
  perfectly) — same as all Claude Code rules
- Hard enforcement (blocking tools, requiring MCPs) stays in
  `settings.json` — profiles only declare what's expected, they don't
  auto-configure `settings.json` in V1
- Profile files must be small (<50 lines) to minimize context cost

---

## Success Metrics

1. A user can switch between `quality` and `fast` profiles and see
   measurably different behavior from `/dev:impl`
2. `/dev:health` correctly reports missing MCPs for the active profile
3. Zero functional regression — commands work identically with no
   profile active (defaults preserved)
